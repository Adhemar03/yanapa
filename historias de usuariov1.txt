Descripción
Como usuario nuevo,
quiero registrarme utilizando mi cuenta de Google,
para acceder rápidamente a Yanapa sin necesidad de crear una contraseña.
Objetivo
Reducir el tiempo de registro del usuario aprovechando el mecanismo de autenticación OAuth 2.0 proporcionado por Google.
Flujo principal
El usuario abre Yanapa.
Selecciona "Continuar con Google".
La aplicación solicita autenticación a Google.
Google verifica las credenciales.
Supabase recibe el token OAuth.
Se crea automáticamente el registro en auth.users.
El trigger crea el perfil en users_schema.perfiles.
El sistema redirecciona al Home.
Flujo alternativo
El usuario cancela el proceso de autenticación.
Google devuelve un error.
No existe conexión a Internet.
La cuenta Google se encuentra restringida.
Precondiciones
El dispositivo debe tener conexión a Internet.
Google OAuth debe estar habilitado.
Supabase Auth debe encontrarse operativo.
Postcondiciones
El usuario queda autenticado.
Existe un registro en auth.users.
Existe un perfil en users_schema.perfiles.
Se genera un JWT válido.
Reglas de negocio
RN-001
El correo electrónico debe ser único.
RN-002
No podrán existir dos perfiles asociados al mismo usuario de Google.
RN-003
Todo usuario registrado mediante Google será inicialmente Cliente.
RN-004
El sistema obtendrá automáticamente:

Fotografía
Correo electrónico

Administrar los diferentes roles que puede desempeñar un usuario dentro del sistema.

Descripción
Como cliente, quiero consultar el estado de mis solicitudes de servicio, para saber si fueron aceptadas, rechazadas, atendidas o finalizadas.
Objetivo
Permitir que el cliente realice seguimiento de sus solicitudes registradas dentro de Yanapa, visualizando el estado actual y los datos principales del servicio.
Flujo principal
El cliente ingresa a “Mis solicitudes”.
El sistema consulta las solicitudes asociadas al cliente.
La aplicación muestra el listado de solicitudes.
El cliente selecciona una solicitud.
El sistema muestra el detalle y estado actual.
El cliente puede ver información del técnico, categoría, fecha y ubicación.
Flujo alternativo
·       El cliente no tiene solicitudes registradas.
·       La solicitud fue cancelada.
·       No existe conexión a Internet.
·       El servidor no responde.
Precondiciones
·       El cliente debe estar autenticado.
·       El cliente debe tener solicitudes registradas para visualizar detalle.
Postcondiciones
·       El cliente visualiza el estado de sus solicitudes.
·       El cliente puede hacer seguimiento al servicio.
Estados posibles de una solicitud
Estado
Descripción
PENDIENTE
La solicitud fue creada y espera respuesta del técnico.
ACEPTADA
El técnico aceptó atender el servicio.
RECHAZADA
El técnico rechazó la solicitud.




EN_PROCESO
El técnico está atendiendo el servicio.
FINALIZADA
El servicio fue concluido.
CANCELADA
El cliente canceló la solicitud.

 
Reglas de negocio
Código
Regla
RN-044
El cliente solo podrá ver sus propias solicitudes.
RN-045
El estado de la solicitud solo podrá cambiar según el flujo permitido.
RN-046
Una solicitud finalizada no podrá volver a estado pendiente.

 
Endpoint
GET /requests/my
Criterios de aceptación
·       Debe mostrar listado de solicitudes del cliente.
·       Debe mostrar estado actual.
·       Debe mostrar técnico asignado.
·       Debe mostrar fecha y categoría.
·       Debe permitir ver detalle de cada solicitud.
·       No debe mostrar solicitudes de otros clientes.

5. Resumen del flujo del cliente
Las historias agregadas cubren el flujo principal del cliente dentro de Yanapa: visualizar categorías, filtrar técnicos por categoría, revisar el perfil del técnico, contactar al técnico por teléfono, realizar la solicitud de servicio y consultar el estado de la solicitud. Este flujo permite que la aplicación móvil responda al objetivo central del proyecto: conectar clientes con técnicos domiciliarios de manera rápida, organizada y trazable.
